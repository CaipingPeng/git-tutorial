### 3. Git教程系列-07-变基.md

**增加"何时不该 rebase"的醒目警告：**

\\\markdown
## ⚠️ Rebase 的黄金法则

**永远不要 rebase 已经推送到公共分支的提交。**

### 什么是"公共分支"？

- ✅ 你的个人功能分支（只有你在用）：**可以 rebase**
- ❌ \main\ / \master\ 分支：**不要 rebase**
- ❌ 团队共同开发的分支：**不要 rebase**
- ❌ 已经有 PR 的分支（除非团队约定允许）：**谨慎 rebase**

### 为什么？

Rebase 会重写提交历史（改变提交哈希）。如果别人基于旧的提交开发，你 rebase 后推送会导致：
- 他们的历史和你的历史分叉
- 合并时产生重复提交
- 团队协作混乱

### 安全的 rebase 场景

\\\ash
# ✅ 场景1：整理自己的功能分支后推送
git switch feature-mine
git rebase main
git push --force-with-lease origin feature-mine

# ✅ 场景2：本地整理提交，还没推送
git rebase -i HEAD~3

# ❌ 场景3：永远不要这样做
git switch main
git rebase feature-something  # 不要 rebase main！
git push --force              # 更不要强推 main！
\\\

记住：**当不确定时，用 merge 而不是 rebase。**

### 4. Git教程系列-09-撤销与恢复.md

**增加 reflog 保留期限说明：**

\\\markdown
## Reflog 的保留期限

\git reflog\ 是 Git 的"后悔药"，但它不是永久的：

| 情况 | 保留时间 |
|-----|---------|
| 可达的提交（有分支指向） | 永久保留 |
| 不可达的提交（通过 reflog 找到） | 默认 **90 天** |
| reflog 记录本身 | 默认 **90 天** |

### 这意味着什么？

如果你：
1. 删除了一个分支
2. 通过 \git reflog\ 找到了提交哈希
3. 但过了 90 天没有恢复它

那么这个提交可能会被 Git 垃圾回收机制删除。

### 保险做法

如果通过 reflog 找到了重要提交，立即给它创建一个分支：

\\\ash
git reflog  # 找到提交哈希，例如 abc1234
git switch -c rescue-重要提交 abc1234
\\\

这样提交就变成"可达"的，不会被删除。

### 6. Git教程系列-06-远程协作.md

**增加 HTTPS vs SSH 的选择指导：**

\\\markdown
## 选择远程协议：HTTPS 还是 SSH？

克隆仓库时，你会看到两种 URL：

\\\ash
# HTTPS
https://github.com/username/repo.git

# SSH
git@github.com:username/repo.git
\\\

### 对比

| 特性 | HTTPS | SSH |
|-----|-------|-----|
| 设置难度 | 简单，无需配置 | 需要生成密钥对 |
| 推送认证 | 每次输入密码（或缓存） | 自动认证 |
| 防火墙友好 | 是（端口443） | 有时被防火墙阻止（端口22） |
| 适合场景 | 新手、临时克隆 | 经常推送、多仓库 |

### 建议

- **新手先用 HTTPS**：简单直接
- **频繁推送后配置 SSH**：避免反复输入密码
- **企业环境**：询问团队规范

### 配置 SSH（可选）

\\\ash
# 1. 生成密钥对
ssh-keygen -t ed25519 -C "your_email@example.com"

# 2. 查看公钥
cat ~/.ssh/id_ed25519.pub

# 3. 在 GitHub/GitLab 设置中添加这个公钥

# 4. 测试连接
ssh -T git@github.com
\\\

详细步骤见各平台官方文档：
- GitHub: https://docs.github.com/cn/authentication
- GitLab: https://docs.gitlab.com/ee/user/ssh.html
- Gitee: https://help.gitee.com/base/account/SSH公钥设置

## 如何选择工作流？

\\\mermaid
flowchart TD
    A[选择团队工作流] --> B{团队规模?}
    B -- 1-2人 --> C[集中式工作流]
    B -- 3-10人 --> D{是否需要代码审查?}
    B -- 10人以上 --> E[GitHub Flow 或 GitLab Flow]
    
    D -- 是 --> F[Feature Branch + PR]
    D -- 否 --> C
    
    E --> G{是否维护多个发布版本?}
    G -- 是 --> H[Gitflow 或 GitLab Flow with Release]
    G -- 否 --> I[GitHub Flow]
    
    C:::simple
    F:::moderate
    H:::complex
    I:::simple
    
    classDef simple fill:#d4edda,stroke:#28a745
    classDef moderate fill:#fff3cd,stroke:#ffc107
    classDef complex fill:#f8d7da,stroke:#dc3545
\\\

### 决策因素

| 问题 | 影响 |
|-----|------|
| 团队是否要求代码审查？ | 是 → 必须用 PR/MR 工作流 |
| 是否持续部署（每次合并就发布）？ | 是 → GitHub Flow；否 → Gitflow |
| 是否同时维护多个版本（如 v1.0, v2.0）？ | 是 → Gitflow |
| 团队对 Git 的熟练度？ | 新手多 → 简单工作流 |
| 发布频率？ | 每周/每天 → GitHub Flow；每月/每季度 → Gitflow |
