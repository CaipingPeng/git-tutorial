# Git 代码评审与 PR 质量

第 8 章讲了 PR 的基本流程。本章更进一步：什么是好 PR？审查者应该看什么？作者如何回应评论？合并前应该确认哪些事？

本章目标：

1. 知道 PR 质量为什么影响团队效率
2. 学会写清楚 PR 标题和描述
3. 建立作者与审查者的职责边界
4. 使用合并前检查清单降低风险

---

## 1. PR 不是把代码扔给别人

一个差 PR 常见特征：

- 标题像“update”“fix bug”一样模糊
- 描述为空
- 一次改动过大，混合功能、重构、样式、顺手改错别字
- 没说明怎么验证
- reviewer 不知道应该重点看哪里

一个好 PR 应该让审查者快速回答：

| 问题 | PR 应该提供的信息 |
|---|---|
| 为什么改？ | 背景、问题或需求 |
| 改了什么？ | 主要改动点 |
| 怎么验证？ | 手动步骤、测试命令或截图 |
| 有什么风险？ | 影响范围、已知限制、兼容性 |

---

## 2. 好 PR 标题

标题要短，但不能空。

不推荐：

```text
update
fix
修改代码
登录
```

推荐：

```text
修复登录表单空密码提示
添加用户资料页头像上传
调整订单列表分页逻辑
```

一个实用格式：

```text
动词 + 对象 + 结果
```

例如：

```text
修复远程同步章节中的 force push 风险说明
```

---

## 3. PR 描述模板

可以使用这个模板：

```markdown
## 背景

为什么需要这个改动？

## 改动

- 改了什么
- 没改什么

## 验证

- [ ] 运行了什么命令
- [ ] 手动检查了什么页面或流程

## 风险

哪些地方可能受影响？有没有需要 reviewer 特别看的地方？
```

如果是小改动，不必写长篇。但至少要说明“为什么”和“怎么验证”。

### PR 元数据怎么用

PR 页面右侧通常会有 reviewer、assignee、label、project、milestone 等字段。它们不是装饰，而是在回答“谁负责下一步”“这个改动属于哪类工作”“它和哪个版本或计划有关”。

| 字段 | 解决什么问题 | 新手建议 |
|---|---|---|
| Reviewer | 谁来审查代码和风险 | 选真正熟悉这块代码的人，不要把全团队都加上 |
| Assignee | 谁负责推动下一步 | 通常保持一个主要负责人，避免大家都以为别人会处理 |
| Label | 当前状态或类型 | 用 `bug`、`docs`、`ready for review`、`needs discussion` 这类标签帮助筛选 |
| Project / Milestone | 属于哪个计划或版本 | 适合有迭代计划、发布节奏或维护路线图的仓库 |
| Linked issue | 背景讨论在哪里 | PR 解决某个 issue 时，在描述里关联它 |

如果你不知道某个字段该填什么，可以先不填，或按团队模板来。元数据的目标是减少沟通成本，不是让 PR 页面看起来更“完整”。

## 4. 小 PR 原则

PR 越大，审查质量越低。

| PR 大小 | 审查效果 |
|---|---|
| 小而聚焦 | 审查者能认真看，反馈具体 |
| 中等但结构清楚 | 可以接受，需要好描述 |
| 巨大且混杂 | 审查者容易只看表面，风险上升 |

拆 PR 的原则：

- 功能改动和纯重构尽量分开
- 文案/格式清理不要混进核心逻辑 PR
- 一个 PR 只解决一个主要问题
- 如果必须大改，用描述告诉 reviewer 先看哪些文件

### 大 PR 怎么拆：一个例子

假设你重写了登录模块：换了校验逻辑、改了组件结构、顺手修了样式、还更新了文档。一个 PR 全包会让 reviewer 无从下手。可以按"能独立成立"的边界拆成几步：

| 顺序 | PR | 内容 | 为什么单独成 PR |
|---|---|---|---|
| 1 | 重构：抽取登录校验函数 | 只移动代码、不改行为 | 纯重构，reviewer 只需确认行为不变 |
| 2 | 功能：新校验逻辑 | 基于第 1 步的新函数改逻辑 | 改动聚焦在行为差异上 |
| 3 | 样式：登录页视觉调整 | 只动 CSS | 不混进逻辑判断 |
| 4 | 文档：更新登录说明 | 只改 README | 文案风险低，可快速过 |

拆分的关键不是"文件多少"，而是"每个 PR 能不能被独立理解和回滚"。如果第 2 步出问题需要 revert，不会把第 1 步的重构也一起撤掉，这才是拆 PR 的价值。

如果改动高度耦合、确实无法拆，至少在描述里用"建议审查顺序"引导 reviewer，例如先看接口契约，再看实现，最后看测试。

## 5. 作者职责

PR 作者不是提交完就结束。

作者应该：

1. 自己先读一遍 diff。
2. 删除无关改动。
3. 写清楚标题、描述和验证步骤。
4. 主动标注风险点。
5. 对评论做明确回应。
6. 修改后不要让 PR 长期挂起。

提交 PR 前可以运行：

```bash
git status
git log --oneline --graph --decorate -5
git diff origin/main...HEAD
```

最后一条能帮你看“这个 PR 相对目标分支到底包含什么”。

## 6. 审查者职责

审查者不是挑刺机器，也不是只看格式。

重点看：

| 维度 | 问题 |
|---|---|
| 正确性 | 这个改动是否解决了问题？有没有明显 bug？ |
| 安全性 | 是否引入凭据泄露、命令注入、权限绕过等风险？ |
| 可维护性 | 命名、结构、边界是否清楚？ |
| 测试/验证 | 作者的验证是否足够？ |
| 范围 | 是否混入无关改动？ |

评论时尽量具体：

不推荐：

```text
这里不行
```

推荐：

```text
这里在输入为空时会跳过校验，是否应该和注册页保持同样的错误提示？
```

### 给出可执行的评论

好的 review 评论不只是指出问题，还让作者知道下一步该做什么。判断标准是：作者读完评论后，能否直接动手，而不需要再追问"那你希望我改成什么样"。

| 评论类型 | 例子 | 效果 |
|---|---|---|
| 只表态，不可执行 | "这写法不好" | 作者不知道往哪改 |
| 指出问题，给方向 | "这里循环里调用了数据库，列表大时会很慢，能否批量查询？" | 作者知道问题在哪、怎么改 |
| 指出问题，给方案 | "建议把校验抽成函数，参数用对象而不是五个布尔值，参考 userValidator.js" | 作者能直接照做或讨论替代方案 |

可执行评论的几个特征：

- 说清"在哪、什么问题、为什么是问题"，而不是只下结论
- 如果有偏好，明确给出方向或参考，而不是让作者猜
- 把"必须改"和"建议讨论"分开，避免作者把所有评论都当成阻塞
- 涉及边界条件时，给出能复现的输入或步骤

如果评论本身需要较长讨论，可以在 PR 里单独开一个 resolved 标记的讨论线程，不要把大段设计争议埋在某一行的行内评论里。

一条好的 review 评论本身就像一个小 PR：有上下文、有建议、可执行。尽量让作者看完就知道改什么、为什么改。

| 评论要素 | 说明 | 例子 |
|---|---|---|
| 指出位置 | 具体到文件和行 | `src/login.js:42` |
| 说明问题 | 描述现象或风险，不只下结论 | "空密码时直接返回 true，会绕过校验" |
| 给出方向 | 提建议而非命令，留余地 | "是否可以和注册页一样先判空再走校验？" |
| 区分严重度 | 让作者知道必须改还是可以讨论 | "阻塞：有安全风险" / "建议：风格统一" |

避免只丢一个"这样不好"或"改一下"。如果评论本身需要作者反复追问才能理解，审查效率会下降，也容易让作者感到被否定。

评论不一定要给出完整代码。能说清"在哪里、什么问题、想往哪个方向走"就足够；具体实现交给作者，往往更符合项目风格。

## 7. 如何回应 review

收到评论时，不要把 review 当成人身否定。PR 审查是在保护项目，不是在证明谁更懂。

常见回应：

| 情况 | 回应方式 |
|---|---|
| 评论正确 | 修改后回复“已调整”，必要时说明怎么改 |
| 不确定 | 追问具体场景或约束 |
| 不同意 | 解释原因，并给出替代方案 |
| 评论超出本 PR 范围 | 说明可另开 issue/PR 处理 |

如果 reviewer 指出真实风险，不要只改表面；要回到问题根因。

## 8. 合并前检查清单

合并前至少确认：

- [ ] PR 目标分支正确
- [ ] reviewer、assignee、label、milestone 等元数据符合团队规则
- [ ] 标题和描述能说明背景
- [ ] 没有无关文件
- [ ] 没有密钥、token、`.env` 等敏感内容
- [ ] CI 或必要测试通过
- [ ] 冲突已解决
- [ ] reviewer 已批准或团队规则允许合并
- [ ] 合并方式符合团队约定
- [ ] PR 分支已经基于足够新的目标分支验证过

如果 PR 已经落后目标分支很多，审查者看到的代码可能无法代表最终合并后的状态。先按团队规则更新分支，再重新看 CI 和关键验证结果。

| 场景 | 建议 |
|---|---|
| 这是作者自己的短生命周期分支 | 可以按团队规则 rebase 到最新 `main` 后再 push |
| 这是多人共同开发或已公开依赖的分支 | 合并 `main` 到该分支，保留共享历史 |

旧的绿色检查不一定能证明新的合并结果安全。分支更新后，至少要确认冲突解决、测试结果和 PR diff 都仍然符合预期。

---

## 9. 合并方式怎么选

| 方式 | 优点 | 代价 | 适合 |
|---|---|---|---|
| Merge commit | 保留真实分支历史 | 历史可能更复杂 | 需要保留上下文的团队 |
| Squash and merge | 主线每个 PR 一个提交 | PR 内部提交细节消失 | 小功能、提交杂乱的 PR |
| Rebase and merge | 主线线性 | 改写分支提交 | 团队熟悉 rebase 且追求线性历史 |

不要因为某种方式“高级”就默认使用。合并方式应该服务团队阅读历史和回滚的需要。

谁来点击“合并”也应该有规则。小团队可以允许作者在 CI 通过、review 批准后自行合并；风险更高的项目可以要求非作者合并，或通过分支保护强制至少一名 reviewer 批准。关键不是谁的权限更大，而是让最终合入主线的人对状态、风险和回滚方案心里有数。

## 10. 本章检查点

1. 一个好 PR 描述至少应该包含哪三类信息？
2. 为什么功能改动和重构最好不要混在同一个 PR？
3. reviewer 评论应该关注哪些维度？
4. 三种合并方式分别适合什么场景？
