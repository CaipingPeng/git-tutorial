# Git 代码评审与 PR 质量

第 8 章讲了 PR 的基本流程。本章更进一步：什么是好 PR？审查者应该看什么？作者如何回应评论？合并前应该确认哪些事？

本章目标：

1. 知道 PR 质量为什么影响团队效率
2. 学会写清楚 PR 标题和描述
3. 建立作者与审查者的职责边界
4. 使用合并前检查清单降低风险

---

## 1. PR 不是把代码扔给别人

一个差 PR 常见特征：

- 标题像“update”“fix bug”一样模糊
- 描述为空
- 一次改动过大，混合功能、重构、样式、顺手改错别字
- 没说明怎么验证
- reviewer 不知道应该重点看哪里

一个好 PR 应该让审查者快速回答：

| 问题 | PR 应该提供的信息 |
|---|---|
| 为什么改？ | 背景、问题或需求 |
| 改了什么？ | 主要改动点 |
| 怎么验证？ | 手动步骤、测试命令或截图 |
| 有什么风险？ | 影响范围、已知限制、兼容性 |

---

## 2. 好 PR 标题

标题要短，但不能空。

不推荐：

```text
update
fix
修改代码
登录
```

推荐：

```text
修复登录表单空密码提示
添加用户资料页头像上传
调整订单列表分页逻辑
```

一个实用格式：

```text
动词 + 对象 + 结果
```

例如：

```text
修复远程同步章节中的 force push 风险说明
```

---

## 3. PR 描述模板

可以使用这个模板：

```markdown
## 背景

为什么需要这个改动？

## 改动

- 改了什么
- 没改什么

## 验证

- [ ] 运行了什么命令
- [ ] 手动检查了什么页面或流程

## 风险

哪些地方可能受影响？有没有需要 reviewer 特别看的地方？
```

如果是小改动，不必写长篇。但至少要说明“为什么”和“怎么验证”。

---

## 4. 小 PR 原则

PR 越大，审查质量越低。

| PR 大小 | 审查效果 |
|---|---|
| 小而聚焦 | 审查者能认真看，反馈具体 |
| 中等但结构清楚 | 可以接受，需要好描述 |
| 巨大且混杂 | 审查者容易只看表面，风险上升 |

拆 PR 的原则：

- 功能改动和纯重构尽量分开
- 文案/格式清理不要混进核心逻辑 PR
- 一个 PR 只解决一个主要问题
- 如果必须大改，用描述告诉 reviewer 先看哪些文件

---

## 5. 作者职责

PR 作者不是提交完就结束。

作者应该：

1. 自己先读一遍 diff。
2. 删除无关改动。
3. 写清楚标题、描述和验证步骤。
4. 主动标注风险点。
5. 对评论做明确回应。
6. 修改后不要让 PR 长期挂起。

提交 PR 前可以运行：

```bash
git status
git log --oneline --graph --decorate -5
git diff origin/main...HEAD
```

最后一条能帮你看“这个 PR 相对目标分支到底包含什么”。

---

## 6. 审查者职责

审查者不是挑刺机器，也不是只看格式。

重点看：

| 维度 | 问题 |
|---|---|
| 正确性 | 这个改动是否解决了问题？有没有明显 bug？ |
| 安全性 | 是否引入凭据泄露、命令注入、权限绕过等风险？ |
| 可维护性 | 命名、结构、边界是否清楚？ |
| 测试/验证 | 作者的验证是否足够？ |
| 范围 | 是否混入无关改动？ |

评论时尽量具体：

不推荐：

```text
这里不行
```

推荐：

```text
这里在输入为空时会跳过校验，是否应该和注册页保持同样的错误提示？
```

---

## 7. 如何回应 review

收到评论时，不要把 review 当成人身否定。PR 审查是在保护项目，不是在证明谁更懂。

常见回应：

| 情况 | 回应方式 |
|---|---|
| 评论正确 | 修改后回复“已调整”，必要时说明怎么改 |
| 不确定 | 追问具体场景或约束 |
| 不同意 | 解释原因，并给出替代方案 |
| 评论超出本 PR 范围 | 说明可另开 issue/PR 处理 |

如果 reviewer 指出真实风险，不要只改表面；要回到问题根因。

---

## 8. 合并前检查清单

合并前至少确认：

- [ ] PR 目标分支正确
- [ ] 标题和描述能说明背景
- [ ] 没有无关文件
- [ ] 没有密钥、token、`.env` 等敏感内容
- [ ] CI 或必要测试通过
- [ ] 冲突已解决
- [ ] reviewer 已批准或团队规则允许合并
- [ ] 合并方式符合团队约定

如果 PR 已经落后目标分支很多，先按团队规则更新分支：merge 或 rebase。

---

## 9. 合并方式怎么选

| 方式 | 优点 | 代价 | 适合 |
|---|---|---|---|
| Merge commit | 保留真实分支历史 | 历史可能更复杂 | 需要保留上下文的团队 |
| Squash and merge | 主线每个 PR 一个提交 | PR 内部提交细节消失 | 小功能、提交杂乱的 PR |
| Rebase and merge | 主线线性 | 改写分支提交 | 团队熟悉 rebase 且追求线性历史 |

不要因为某种方式“高级”就默认使用。合并方式应该服务团队阅读历史和回滚的需要。

---

## 10. 本章检查点

1. 一个好 PR 描述至少应该包含哪三类信息？
2. 为什么功能改动和重构最好不要混在同一个 PR？
3. reviewer 评论应该关注哪些维度？
4. 三种合并方式分别适合什么场景？
