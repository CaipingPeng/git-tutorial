## 三、发现的优化机会

### 3.1 高优先级改进（建议实施）

#### 改进1：增强第3章 git commit -a 说明

**当前内容**：
`markdown
-a 的意思是 "all tracked files"。注意它**不会 add 新文件**（untracked）
`

**建议改为**：
`markdown
-a 的意思是 "all tracked files"。它有两个重要限制：

1. **不会 add 新文件**（untracked files）
2. **不会 add 被 .gitignore 忽略的文件**

这意味着如果你：
- 创建了新文件 → 必须先 git add 新文件
- 想精确控制暂存哪些改动 → 不能用 -a

**什么时候用 -a**：
| 场景 | 用不用 -a |
|---|---|
| 只改了已有文件，想全部提交 | ✅ 可以用 git commit -a |
| 有新文件要提交 | ❌ 必须先 git add 新文件 |
| 想只提交部分改动 | ❌ 不能用 -a，要手动选择 |

**推荐习惯**：新手阶段建议总是先 git add，再 git commit，这样更容易理解暂存区的作用。
`

**理由**：原文虽然正确，但新手容易误用，增强说明可避免困惑。

#### 改进2：补充第1章"快照vs差异"的视觉对比

**当前内容**：有图片但文字说明可以更直观

**建议增加**：
`markdown
### 快照模型的实际意义

**传统差异式**：
\\\
版本1：完整项目
版本2：版本1 + 补丁A
版本3：版本2 + 补丁B
\\\

要恢复版本3，必须：版本1 → 应用补丁A → 应用补丁B

**Git 快照式**：
\\\
版本1：完整快照
版本2：完整快照
版本3：完整快照
\\\

要恢复版本3，直接：取出版本3的快照

**这带来什么好处？**

| 操作 | 差异式 | 快照式（Git） |
|-----|-------|-------------|
| 切换到任意版本 | 需要应用一串补丁 | 直接取出快照 |
| 创建分支 | 需要复制文件 | 只需创建指针 |
| 合并分支 | 复杂的补丁合并 | 三方快照合并 |
| 检出速度 | 慢 | 快 |

> **注意**：Git 内部使用压缩和增量存储优化空间，但对用户来说，可以把每次提交理解为完整快照。
`
