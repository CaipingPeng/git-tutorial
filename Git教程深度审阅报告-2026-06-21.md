|------|---------|---------|------|
| git add | 将文件加入暂存区 | Add file contents to the index | ✅ 准确 |
| git commit | 保存暂存区到本地仓库 | Record changes to the repository | ✅ 准确 |
| git reset --soft | 移动 HEAD，保留暂存区和工作目录 | 官方确认此行为 | ✅ 准确 |

### 1.2 需要澄清的技术点

#### ⚠️ 第9章：reset 的工作目录状态描述

**当前描述**（第9章）：
> "git reset --mixed HEAD~1 会把改动保留在工作目录"

**问题**：这个描述是准确的，但可能让新手误解为"会主动把暂存区的改动移回工作目录"。

**官方文档原文**：
> git reset [--mixed] <commit> resets the index but not the working tree

**建议改进**：
\\\markdown
## reset --mixed 的准确行为

\git reset --mixed HEAD~1\ 会：
1. ✅ 移动 HEAD 和当前分支指针到 HEAD~1
2. ✅ 重置暂存区，使其匹配 HEAD~1
3. ✅ **不改变工作目录**（文件内容保持不变）

**结果**：之前提交的改动会表现为"未暂存的修改"（因为暂存区被重置了，但工作目录还保留着你的改动）

| reset 模式 | HEAD | 暂存区 | 工作目录 |
|-----------|------|-------|---------|
| --soft | 移动 | 不变 | 不变 |
| --mixed (默认) | 移动 | 重置 | 不变 |
| --hard | 移动 | 重置 | 重置 |
\\\

#### ⚠️ 第6章：远程跟踪分支的更新时机

**当前描述**：
> "远程跟踪分支会在 git fetch 时自动更新"

**建议补充**：
\\\markdown
### 远程跟踪分支何时更新？

远程跟踪分支（如 \origin/main\）**只在以下操作时更新**：
- \git fetch\ - 明确拉取远程更新
- \git pull\ - 内部会先执行 fetch
- \git push\ - 推送成功后更新对应的远程跟踪分支
- \git clone\ - 首次克隆时创建

**不会自动更新的情况**：
- ❌ 切换分支时
- ❌ 提交时
- ❌ 别人推送到远程后（你的本地 \origin/main\ 不会自动感知）

**查看远程分支最新状态**：
\\\ash
git fetch origin          # 先更新远程跟踪分支
git log origin/main       # 再查看远程分支历史
\\\
\\\

#### ⚠️ 第7章：Rebase 黄金法则的准确表述

**当前描述**：
> "不要 rebase 已经推送到公共仓库的提交"

**建议改进**（更准确）：
\\\markdown
### Rebase 黄金法则

**核心原则**：不要 rebase 其他人可能基于其工作的提交。

具体场景：

| 场景 | 能否 rebase | 原因 |
|------|-----------|------|
| 自己的功能分支，还没推送 | ✅ 安全 | 没人依赖这些提交 |
| 自己的功能分支，已推送但只有你在用 | ✅ 相对安全 | 需要 \--force-with-lease\ |
| 团队共享的功能分支 | ⚠️ 需团队共识 | 可能打乱别人的工作 |
| main/master 等主分支 | ❌ 绝对不要 | 会破坏所有人的历史 |
| 已合并到 main 的提交 | ❌ 绝对不要 | 历史已经公开 |

**为什么不能 rebase 公共历史？**

Rebase 会改变提交的哈希值（因为父提交变了）。如果别人已经基于旧哈希工作，他们的历史会断裂。

**例外情况**：
有些团队约定"功能分支可以 rebase main，但 main 不能 rebase"。这需要团队明确共识。

### 3.1 Git 2.23+ 新命令说明

**背景**：Git 2.23 (2019年8月) 引入了 \git switch\ 和 \git restore\ 来替代 \git checkout\ 的部分功能。

**当前教程处理**：
- ✅ 已使用 \git switch\ 作为主要的分支切换命令
- ✅ 已使用 \git restore\ 作为文件恢复命令
- ❌ 缺少新旧命令对照表

**建议补充**（在第3章或附录）：

\\\markdown
## Git 命令演进：checkout vs switch/restore

### 为什么有两套命令？

\git checkout\ 功能太多，既能切换分支，又能恢复文件，容易混淆。Git 2.23 引入了更清晰的命令：

| 旧命令 (仍可用) | 新命令 (推荐) | 作用 |
|---------------|-------------|------|
| \git checkout 分支名\ | \git switch 分支名\ | 切换分支 |
| \git checkout -b 新分支\ | \git switch -c 新分支\ | 创建并切换分支 |
| \git checkout -- 文件\ | \git restore 文件\ | 恢复文件到最后提交状态 |
| \git checkout HEAD~1 文件\ | \git restore --source=HEAD~1 文件\ | 从特定提交恢复文件 |

### 我应该用哪套？

**推荐新手用新命令**：
- \git switch\ 专注分支操作，语义清晰
- \git restore\ 专注文件恢复，不会误操作分支

**团队协作考虑**：
- 如果团队用 Git < 2.23，建议统一用旧命令
- 如果团队已升级，优先用新命令

**本教程选择**：
主要使用新命令，首次出现时会标注旧命令等价写法。

### 3.2 常见误解澄清专题

建议增加一个独立文档或在 troubleshooting.md 中增加章节：

\\\markdown
## Git 常见误解澄清

### 误解1：git pull = 下载代码
❌ **错误理解**：\git pull\ 只是把远程代码下载到本地

✅ **准确理解**：\git pull = git fetch + git merge\
- 会自动合并远程分支到当前分支
- 可能产生合并冲突
- 如果只想查看远程更新，用 \git fetch\

---

### 误解2：删除分支会删除提交
❌ **错误理解**：\git branch -d feature\ 会删除这个分支上的所有提交

✅ **准确理解**：
- 删除分支只是删除"指针"（分支名）
- 如果提交已合并到其他分支，提交仍然存在
- 如果提交没合并，Git 会警告并拒绝删除（除非用 \-D\ 强制）
- 即使强制删除，提交在 reflog 中仍可找回（30天内）

---

### 误解3：git commit 会上传到 GitHub
❌ **错误理解**：\git commit\ 后代码就到 GitHub 了

✅ **准确理解**：
- \git commit\ 只保存到**本地仓库**
- 需要 \git push\ 才能上传到远程（如 GitHub）
- 断网时仍可 commit，联网后再 push

---

### 误解4：.gitignore 能忽略已提交的文件
❌ **错误理解**：把文件加到 \.gitignore\ 就能从 Git 历史中移除

✅ **准确理解**：
- \.gitignore\ 只对**未跟踪文件**生效
- 已提交的文件需要先 \git rm --cached 文件名\
- 这只能让文件不再被跟踪，历史记录仍然存在

---

### 误解5：rebase 比 merge 更好
❌ **错误理解**：rebase 能让历史更干净，所以总是用 rebase

✅ **准确理解**：
- Merge 保留真实历史，适合记录"何时合并"
- Rebase 让历史线性，但会改变提交哈希
- **公共分支用 merge，私有分支可 rebase**
- 团队规范比"哪个更好"更重要

---

### 误解6：冲突是 Git 的错误
❌ **错误理解**：出现冲突说明我操作错了

✅ **准确理解**：
- 冲突是**正常现象**，不是错误
- 说明同一文件的同一位置被两个人修改了
- Git 无法自动决定保留谁的，需要人工判断
- 解决冲突是协作开发的必备技能

---

### 误解7：HEAD 是最新提交
❌ **错误理解**：HEAD 永远指向最新的提交

✅ **准确理解**：
- HEAD 指向**当前所在的提交**
- 通常情况下 HEAD -> 当前分支 -> 最新提交
- 但可以 checkout 到历史提交（detached HEAD）
- 可以切换到其他分支（HEAD 会移动）

git reflog

# 2. 给那个提交打个临时分支（保险）
git branch backup-临时 <出问题前的哈希>

# 3. 再决定是 reset 还是其他操作
git reset --hard backup-临时
