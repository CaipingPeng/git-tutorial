[p] git-rebase - Reapply commits on top of another base tip

[pre] git rebase[-i | --interactive] [<options>] [--exec <cmd>]
	[--onto <newbase> | --keep-base] [<upstream> [<branch>]]git rebase[-i | --interactive] [<options>] [--exec <cmd>] [--onto <newbase>]

[h2] DESCRIPTION

[p] Transplant a series of commits onto a different starting point.
You can also usegitrebaseto reorder or combine commits: see INTERACTIVE
MODE below for how to do that.

[p] For example, imagine that you have been working on thetopicbranch in this
history, and you want to "catch up" to the work done on themasterbranch.

by runninggitrebasemasterwhile thetopicbranch is checked out.  If you
want to rebasetopicwhile on another branch,gitrebasemastertopicis a
shortcut forgit checkout topic && git rebase master.

[h2] TRANSPLANTING A TOPIC BRANCH WITH --ONTO

[p] Here is how you would transplant a topic branch based on one
branch to another, to pretend that you forked the topic branch
from the latter branch, usingrebase--onto.

[p] First let’s assume yourtopicis based on branchnext.
For example, a feature developed intopicdepends on some
functionality which is found innext.

[pre] o---o---o---o---o  master
         \
          o---o---o---o---o  next
                           \
                            o---o---o  topic

[p] We want to maketopicforked from branchmaster; for example,
because the functionality on whichtopicdepends was merged into the
more stablemasterbranch. We want our tree to look like this:

[pre] o---o---o---o---o  master
        |            \
        |             o'--o'--o'  topic
         \
          o---o---o---o---o  next

[p] We can get this using the following command:

[pre] git rebase --onto master next topic

[p] Another example of --onto option is to rebase part of a
branch.  If we have the following situation:

[pre] H---I---J topicB
                           /
                  E---F---G  topicA
                 /
    A---B---C---D  master

[p] then the command

[pre] git rebase --onto master topicA topicB

[p] would result in:

[pre] H'--I'--J'  topicB
                /
                | E---F---G  topicA
                |/
    A---B---C---D  master

[p] This is useful when topicB does not depend on topicA.

[p] A range of commits could also be removed with rebase.  If we have
the following situation:

[pre] E---F---G---H---I---J  topicA

[p] then the command

[pre] git rebase --onto topicA~5 topicA~3 topicA

[p] would result in the removal of commits F and G:
