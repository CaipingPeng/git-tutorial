[p] git-range-diff - Compare two commit ranges (e.g. two versions of a branch)

[pre] git range-diff[--color=[<when>]] [--no-color] [<diff-options>]
	[--no-dual-color] [--creation-factor=<factor>]
	[--left-only | --right-only] [--diff-merges=<format>]
	[--remerge-diff]
	( <range1> <range2> | <rev1>…​<rev2> | <base> <rev1> <rev2> )
	[[--] <path>…​]

[h2] DESCRIPTION

[p] This command shows the differences between two versions of a patch
series, or more generally, two commit ranges (ignoring merge commits).

[p] In the presence of<path>arguments, these commit ranges are limited
accordingly.

[p] To that end, it first finds pairs of commits from both commit ranges
that correspond with each other. Two commits are said to correspond when
the diff between their patches (i.e. the author information, the commit
message and the commit diff) is reasonably small compared to the
patches' size. See ``Algorithm`` below for details.

[p] Finally, the list of matching commits is shown in the order of the
second commit range, with unmatched commits being inserted just after
all of their ancestors have been shown.

[p] There are three ways to specify the commit ranges:

[li] <range1><range2>: Either commit range can be of the form<base>..<rev>,<rev>^!or<rev>^-<n>. SeeSPECIFYINGRANGESingitrevisions[7]for more details.

[p] <range1><range2>: Either commit range can be of the form<base>..<rev>,<rev>^!or<rev>^-<n>. SeeSPECIFYINGRANGESingitrevisions[7]for more details.

[li] <rev1>...<rev2>. This is equivalent to<rev2>..<rev1><rev1>..<rev2>.

[p] <rev1>...<rev2>. This is equivalent to<rev2>..<rev1><rev1>..<rev2>.

[li] <base><rev1><rev2>: This is equivalent to<base>..<rev1><base>..<rev2>.

[p] <base><rev1><rev2>: This is equivalent to<base>..<rev1><base>..<rev2>.

[p] When the commit diffs differ,gitrange-diffrecreates the
original diffs' coloring, and adds outer -/+ diff markers with
thebackgroundbeing red/green to make it easier to see e.g.
when there was a change in what exact lines were added.

[p] Additionally, the commit diff lines that are only present in the first commit
range are shown "dimmed" (this can be overridden using thecolor.diff.<slot>config setting where<slot>is one ofcontextDimmed,oldDimmedandnewDimmed), and the commit diff lines that are only present in the second
commit range are shown in bold (which can be overridden using the config
settingscolor.diff.<slot>with<slot>being one ofcontextBold,oldBoldornewBold).

[p] gitrange-diffalso accepts the regular diff options (seegit-diff[1]), most notably the--color=[<when>]and--no-coloroptions. These options are used when generating the "diff

[h2] OUTPUT STABILITY

[p] The output of therange-diffcommand is subject to change. It is
intended to be human-readable porcelain output, not something that can
be used across versions of Git to get a textually stablerange-diff(as opposed to something like the--stableoption togit-patch-id[1]). There’s also no equivalent ofgit-apply[1]forrange-diff, the output is not intended to
be machine-readable.

[p] This is particularly true when passing in diff options. Currently some
options like--statcan, as an emergent effect, produce output
that’s quite useless in the context ofrange-diff. Future versions
ofrange-diffmay learn to interpret such options in a manner
specific torange-diff(e.g. for--statproducing human-readable
output which summarizes how the diffstat changed).
