[p] git-format-patch - Prepare patches for e-mail submission

[pre] git format-patch[-k] [(-o|--output-directory) <dir> | --stdout]
		   [--no-thread | --thread[=<style>]]
		   [(--attach|--inline)[=<boundary>] | --no-attach]
		   [-s | --signoff]
		   [--signature=<signature> | --no-signature]
		   [--signature-file=<file>]
		   [-n | --numbered | -N | --no-numbered]
		   [--start-number <n>] [--numbered-files]
		   [--in-reply-to=<message-id>] [--suffix=.<sfx>]
		   [--ignore-if-in-upstream] [--always]
		   [--cover-from-description=<mode>]
		   [--rfc[=<rfc>]

[h2] DESCRIPTION

[p] Prepare each non-merge commit with its "patch" in
one "message" per commit, formatted to resemble a UNIX mailbox.
The output of this command is convenient for e-mail submission or
for use withgit am.

[p] A "message" generated by the command consists of three parts:

[li] A brief metadata header that begins withFrom<commit>with a fixedMonSep1700:00:002001datestamp to help programs
like "file(1)" to recognize that the file is an output from this
command, fields that record the author identity, the author date,
and the title of the change (taken from the first paragraph of the
commit log message).

[p] A brief metadata header that begins withFrom<commit>with a fixedMonSep1700:00:002001datestamp to help programs
like "file(1)" to recognize that the file is an output from this
command, fields that record the author identity, the author date,
and the title of the change (taken from the first paragraph of the
commit log message).

[li] The second and subsequent paragraphs of the commit log message.

[p] The second and subsequent paragraphs of the commit log message.

[li] The "patch", which is the "diff -p --stat" output (seegit-diff[1]) between the commit and its parent.

[p] The "patch", which is the "diff -p --stat" output (seegit-diff[1]) between the commit and its parent.

[p] The log message and the patch are separated by a line with a
three-dash line.

[p] There are two ways to specify which commits to operate on.

[li] A single commit, <since>, specifies that the commits leading
to the tip of the current branch that are not in the history
that leads to the <since> to be output.

[p] A single commit, <since>, specifies that the commits leading
to the tip of the current branch that are not in the history
that leads to the <since> to be output.

[li] Generic <revision-range> expression (see "SPECIFYING
REVISIONS" section ingitrevisions[7]) means the
commits in the specified range.

[p] Generic <revision-range> expression (see "SPECIFYING
REVISIONS" section ingitrevisions[7]) means the
commits in the specified range.

[p] The first rule takes precedence in the case of a single<commit>.  To
apply the second rule, i.e., format everything since the beginning of
history up until<commit>, use the--rootoption:gitformat-patch--root<commit>.  If you want to format only<commit>itself, you
can do this withgitformat-patch-1<commit>.

[p] By default, each output file is numbered sequentially from 1, and uses the
first line of the commit message (massaged for pathname safety) as
the filename. With the--numbered-filesoption, the output file names
will only be numbers, without the first line of the commit appended.
The names of the output files are printed to standard
output, unless the--stdoutoption is specified.

[code] --numbered-files

[p] If-ois specified, output files are created in <dir>.  Otherwise
they are created in the current working directory. The default path
can be set with theformat.outputDirectoryconfiguration option.
The-ooption takes precedence overformat.outputDirectory.
To store patches in the current working directory even whenformat.outputDirectorypoints elsewhere, use-o.. All directory
components will be created.

[p] By default, the subject of a single patch is "[PATCH] " followed by
the concatenation of lines from the commit message up to the first blank
line (see the DISCUSSION section ofgit-commit[1]).

[p] When multiple patches are output, the subject prefix will instead be
"[PATCH n/m] ".  To force 1/1 to be added for a single patch, use-n.
To omit patch numbers from the subject, use-N.

[p] If given--thread,git-format-patchwill generateIn-Reply-ToandReferencesheaders to make the second and subsequent patch mails appear
as replies to the first mail; this also generates aMessage-IDheader to
reference.
