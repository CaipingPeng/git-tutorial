[p] git-cherry-pick - Apply the changes introduced by some existing commits

[pre] git cherry-pick[--edit] [-n] [-m <parent-number>] [-s] [-x] [--ff]
		  [-S[<keyid>]]<commit>…​git cherry-pick(--continue | --skip | --abort | --quit)

[h2] DESCRIPTION

[p] Given one or more existing commits, apply the change each one
introduces, recording a new commit for each.  This requires your
working tree to be clean (no modifications from the HEAD commit).

[p] When it is not obvious how to apply a change, the following
happens:

[li] The current branch andHEADpointer stay at the last commit
successfully made.

[p] The current branch andHEADpointer stay at the last commit
successfully made.

[li] TheCHERRY_PICK_HEADref is set to point at the commit that
introduced the change that is difficult to apply.

[p] TheCHERRY_PICK_HEADref is set to point at the commit that
introduced the change that is difficult to apply.

[p] Usually you cannot cherry-pick a merge because you do not know which
side of the merge should be considered the mainline.  This
option specifies the parent number (starting from 1) of
the mainline and allows cherry-pick to replay the change

[p] Usually the command automatically creates a sequence of commits.
This flag applies the changes necessary to cherry-pick
each named commit to your working tree and the index,
without making any commit.  In addition, when this
option is used, your index does not have to match the
HEAD commit.  The cherry-pick is done against the
beginning state of your index.

[p] This is useful when cherry-picking more than one commits'
